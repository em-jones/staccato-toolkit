# Design: Adopt gRPC, Protobuf, and Code Generation Tools

## Context and problem statement

The OpenSpec platform uses undocumented RPC and code generation technologies (gRPC v1.78.0, protobuf, genqlient v0.8.1, gqlparser/v2) across service definitions and tooling. Without documented adoption patterns, usage rules, and code generation practices, maintainers lack clear guidance on schema management, service boundaries, and code generation strategy. This leads to inconsistent implementations, increased onboarding friction, and potential maintenance issues as the codebase evolves.

This design establishes principled patterns for:
1. Defining gRPC services consistently across the platform
2. Managing protobuf schema evolution and backward compatibility
3. Orchestrating code generation (proto → Go, GraphQL schema → genqlient clients)
4. Integrating gqlparser into analysis and generation workflows

## Decision criteria

This design achieves:
- **Consistency**: Unified patterns for all gRPC service definitions and code generation workflows (40%)
- **Maintainability**: Clear ownership, review processes, and documentation for schema evolution (30%)
- **Safety**: Backward compatibility guarantees and preventing accidental breaking changes (20%)
- **Automation**: Build-time validation and automatic code generation tied to schema sources (10%)

Explicitly excludes:
- Implementation of specific service definitions (deferred to feature work per service)
- Deployment orchestration or service registry integration
- Performance optimization of generated code (covered in future operational phases)
- GraphQL schema best practices (separate concern from code generation)

## Considered options

### Option A: No standards — each team adopts tools independently

Proto files, genqlient configs, and code generation scripts are left to individual teams with minimal guidance. Services use different versioning schemes and packaging structures.

**Rejection**: Creates fragmentation, onboarding burden, and risk of incompatible schema evolution across service boundaries.

### Option B: Centralized code generation (monorepo style)

All proto and GraphQL files live in a single central location with shared build rules. Services depend on generated code packages.

**Rejection**: Limits modularity, creates tight coupling between service definitions and consumers, and doesn't match the distributed nature of OpenSpec's architecture.

### Option C: Distributed with documented standards (SELECTED)

Each service owns its proto and GraphQL definitions with clear organization conventions. Standards define versioning, backward compatibility, review processes, and code generation tooling. Teams follow the documented patterns and share code generation configurations.

**Rationale**: Balances autonomy with consistency, supports modular services, aligns with OpenSpec's distributed architecture, and enables effective code review and schema governance.

## Decision outcome

**Adopt distributed gRPC service definitions with centralized tooling and governance standards.**

**Key technical choices**:

1. **Proto file organization**: Services own their proto definitions organized by domain under `proto/` directories using package versioning (e.g., `package user.v1`).

2. **Schema versioning strategy**: Semantic versioning with major.minor only (no patch versions for protos). Breaking changes require new major versions. Backward compatibility is maintained within versions.

3. **Code generation integration**: Protoc and genqlient are configured centrally in `devbox.json` / Makefile / dagger modules. Generation is repeatable and produces committed code artifacts for review.

4. **Backward compatibility guarantee**: Schema changes within a version must allow old and new clients to coexist for at least one major release cycle.

5. **Review process**: All proto changes go through code review with checks for breaking changes, field numbering consistency, and naming conventions.

6. **Generated code safety**: All generated code is marked with autogenerated comments, linted/formatted like handwritten code, and protected from manual edits via pre-commit hooks.

## Risks / trade-offs

- **Risk**: Schema drift between developers if generation isn't reproducible → **Mitigation**: Pinned tool versions in configuration, documented generation process, CI validation of committed vs. generated code
- **Risk**: Breaking changes leak if review process is bypassed → **Mitigation**: Automated checks in pre-commit hooks, clear deprecation timelines, documentation of breaking change policies
- **Trade-off**: Distributing proto definitions increases coordination overhead vs. centralized monorepo → Gain: Modular services, independent deployment, faster iteration per service
- **Risk**: Large generated code diffs make PR review difficult → **Mitigation**: Separate reviews of schema changes from generated code, clear generation tooling documentation

## Migration plan

1. **Phase 1 — Establish standards (this change)**: Document patterns in specs and create usage rules
2. **Phase 2 — Audit existing code**: Catalog current proto definitions and code generation practices
3. **Phase 3 — Tool configuration**: Establish centralized tool versions and generation configs
4. **Phase 4 — Gradual migration**: Migrate services to documented patterns incrementally
5. **Phase 5 — Enforcement**: Enable pre-commit hooks and linting to enforce standards

## Confirmation

How to verify this design is met:

- **Code review checklist**: Proto PRs include manual review of breaking changes and field numbering
- **Metrics**: % of services following naming conventions, generation reproducibility rate
- **Test cases**: CI validates generated code matches committed version, pre-commit hooks prevent manual edits
- **Acceptance criteria**: All new service definitions use documented patterns; review process is in place

## Open questions

- Should gRPC descriptor reflection be enabled in all services or only where explicitly needed?
- What is the deprecation timeline for old proto versions (1 release cycle vs. 2)?
- Should genqlient configuration be shared via a common module or duplicated per service?
