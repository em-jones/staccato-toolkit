## Requirements
### Requirement: Unified code generation tooling configuration

The system SHALL centralize configuration for all code generation tools to ensure consistency and reduce maintenance burden.

All code generation tool versions (protoc, genqlient, gqlparser plugins) SHALL be defined in project configuration (e.g., `devbox.json`, Makefile, or dagger modules). Generation scripts or build rules SHALL use these pinned versions. Tool configuration SHALL be version-controlled and reviewed alongside code changes.

#### Scenario: Configuring genqlient for GraphQL client generation

- **WHEN** setting up code generation infrastructure
- **THEN** genqlient v0.8.1 is specified in configuration, the configuration is committed, and generation can be reproduced locally

#### Scenario: Updating code generation tool version

- **WHEN** a new version of genqlient is available
- **THEN** version is updated in configuration, generated code is regenerated, and changes are reviewed for breaking differences

### Requirement: Proto-to-Go code generation pipeline

The system SHALL automatically generate Go code from protobuf definitions with consistent gRPC service stubs and message types.

The build process SHALL invoke protoc with appropriate plugins to generate Go code in designated output directories. Generated code SHALL be committed to version control (enabling diff review and tracking). Generation SHALL be repeatable and produce identical output across developers and CI. gRPC service interfaces SHALL be generated alongside message types using standard Go protoc plugins.

#### Scenario: Building service with proto definitions

- **WHEN** running the build process after modifying proto files
- **THEN** protoc generates Go code, interfaces, and stubs; all generated files are committed; subsequent builds produce identical output

#### Scenario: Debugging generated code differences

- **WHEN** a developer's generated code differs from committed version
- **THEN** they can reproduce generation locally using same configuration, identify drift, and synchronize

### Requirement: GraphQL client generation with genqlient

The system SHALL use genqlient to generate strongly-typed GraphQL clients from GraphQL schema definitions.

genqlient v0.8.1 SHALL be configured to read GraphQL schema files and query definitions. Generated client packages SHALL provide type-safe query methods. Generated code SHALL be committed and included in version control. Build pipeline SHALL regenerate and validate client code matches schema versions.

#### Scenario: Adding new GraphQL query

- **WHEN** adding a new GraphQL query definition to the codebase
- **THEN** genqlient generates corresponding Go types and query methods, generated code is committed, and clients can use the typed methods

#### Scenario: Updating GraphQL schema

- **WHEN** backend GraphQL schema changes
- **THEN** genqlient regenerates clients, compile errors highlight affected queries, developer updates queries to match schema, and changes are reviewed

### Requirement: gqlparser integration for code analysis

The system SHALL use gqlparser for parsing and analyzing GraphQL definitions in automation tools.

gqlparser v2 SHALL be integrated into code analysis or generation tools that inspect GraphQL schemas. Parser configuration SHALL match the GraphQL version used in the project. gqlparser dependencies SHALL be managed alongside other tool dependencies.

#### Scenario: Analyzing GraphQL schema for validation

- **WHEN** a tool needs to validate GraphQL schema files
- **THEN** gqlparser parses the files, reports syntax errors or inconsistencies, and provides detailed error messages

#### Scenario: Code generation tool using gqlparser

- **WHEN** building custom tooling that analyzes GraphQL queries
- **THEN** gqlparser is used to parse queries, and dependencies are managed with other project tools

### Requirement: Generated code quality and safety

The system SHALL enforce quality standards for all generated code and prevent manual modifications that would be overwritten.

Generated code SHALL be clearly marked with autogenerated comments preventing manual editing. Linting and formatting rules SHALL apply to generated code. Code review processes SHALL verify that generation matches schema versions. Manual edits to generated files SHALL be prevented through warnings or tooling (e.g., read-only files or pre-commit hooks).

#### Scenario: Adding safety check for generated code

- **WHEN** a developer attempts to manually edit a generated file
- **THEN** a pre-commit hook or linter warning alerts them that changes will be overwritten on next generation

#### Scenario: Reviewing generated code differences

- **WHEN** generated code is committed after schema changes
- **THEN** code review includes verification that diffs match schema changes and that tool versions are consistent with configuration

### Requirement: Build artifact lifecycle for generated code

The system SHALL manage the lifecycle of generated code artifacts including creation, storage, and cleanup.

Generated code directories SHALL be clearly marked in build configuration. Build artifacts SHALL be cleaned on `make clean` or equivalent. CI pipelines SHALL regenerate code from source definitions on each build. Generated code SHALL not be committed to releases but instead generated during deployment or runtime as appropriate.

#### Scenario: Cleaning generated artifacts

- **WHEN** running `make clean` or equivalent
- **THEN** generated code directories are deleted, subsequent builds regenerate from source

#### Scenario: Validating generated code in CI

- **WHEN** CI runs on a commit
- **THEN** code generation is executed, results are compared against committed code, and any unexpected differences cause the build to fail
